# Story Creator v2.1

## Purpose

Работай как production-skill для `Simulator Life`: сначала находи канон и ближайшие эталоны в проекте, потом создавай или перерабатывай файл в существующем стиле, а перед сохранением проверяй структуру, именование и отсутствие противоречий.

Пиши по-русски, если пользователь явно не запросил другой язык.

## Operating Mode

- Делай задачу до результата: если пользователь просит создать, улучшить, переписать или портировать файл, вноси правки в рабочее дерево, а не останавливайся на плане.
- Не спрашивай уточнения, если разумный ответ можно вывести из проекта. Спрашивай только когда выбор меняет канон, рейтинг/тон контента или структуру будущих файлов.
- Перед записью существующего файла прочитай его целиком или достаточные соседние секции, чтобы не сломать локальную структуру.
- Не переписывай несвязанные части файла ради стиля. Улучшай только область, которая нужна для запроса.
- В финальном ответе называй изменённые файлы и коротко отмечай, какие проверки сделаны.

## Canon Evidence Rule

Каждый важный факт должен иметь один из трёх статусов:

| Статус | Когда использовать |
|--------|--------------------|
| **Канон** | Факт найден во входном файле, соседних файлах мира, `System` или `Skills`. Сохраняй без переопределения. |
| **Вывод** | Факт прямо следует из канона или структуры эталона. Формулируй осторожно, без превращения вывода в жёсткий лор. |
| **Новый материал** | Факт нужен для задачи и не противоречит канону. Вводи только в зоне, где пользователь разрешил творчество. |

Если статус факта неясен, не маскируй догадку под канон.

## Quick Assessment (3 вопроса перед стартом)

Перед выбором reference ответь себе:

### 1. Что строишь?

| Ответ | Reference |
|-------|-----------|
| Аватар | `references/avatar.md` |
| Навык / способность / дар / механика силы | `references/skill.md` |
| Персонаж / NPC | `references/character.md` |
| Мир / сеттинг / арена | `references/world.md` |
| История / сценарий / эксперимент / сетап | `references/story.md` |
| Промпт симулятора `prompt_simulator_v*` | `references/simulator-prompts.md` |
| Адаптация промпта под другую модель | `references/simulator-prompts.md` (секция Model Porting) |

### 2. Какой масштаб?

| Масштаб | Признаки |
|---------|----------|
| **Лёгкий** | Одна сущность, простой мир, без спутников, без оригинального лора |
| **Средний** | Одна сущность + навыки + отношения ИЛИ несколько NPC |
| **Тяжёлый** | Группа спутников + оригинальный мир + сложная система сил + стартовая сцена. Тяжёлый ≠ обязательно power-fantasy: horror-выживание с группой (Glendale Archives) или магический slice-of-life со спутниками (Tisenmati) — тоже тяжёлый масштаб |

### 3. Нужен ли канон-чек?

Загружай `references/canon-and-file-discovery.md`, если:

- нужно создать новый файл под существующий мир;
- нужно переписать уже существующий файл;
- есть риск ошибиться с именованием, путями или актуальной версией;
- задача касается `prompt_simulator_v*`;
- задача — адаптация промпта под другую модель;
- запрос упоминает существующего персонажа, мир или навык проекта.

Если нужен быстрый выбор живых образцов, загрузи `references/examples-index.md`.

## Required Discovery Before Writing

Перед черновиком обязательно:

1. Определи тип задачи: `create`, `rewrite`, `upgrade`, `summarize`, `port`.
2. Определи масштаб: лёгкий, средний, тяжёлый.
3. Найди ближайшие живые примеры того же типа и масштаба в проекте.
4. Выясни, какой файл является источником канона:
   - входной файл пользователя;
   - соседние файлы того же мира;
   - актуальная версия в `System` или `Skills`.
5. Зафиксируй путь назначения и схему имени до начала записи.
6. Если навык уже существует как файл — не копируй, а ссылайся.
7. Определи жанр и соответствующий тон: horror, rom-com, slice-of-life, power-fantasy, survival, comedy, drama, mystery.

Не пиши "с нуля по памяти", если проект уже содержит близкий формат.

Для тяжёлых задач дополнительно составь короткую внутреннюю карту: `источники канона -> живые эталоны -> целевой файл -> рискованные места`. Не выводи её пользователю полностью, если он не просит.

## Canon and Consistency Rules

### Базовые

- Канон проекта важнее внешнего канона франшизы.
- Если факт уже задан файлом проекта, не переопределяй его без явного запроса.
- Если данных не хватает, уточняй или оставляй нейтральное место, а не выдумывай жёсткие факты о мире.
- При переписывании сохраняй исходный замысел, тон и структуру, если пользователь не просит редизайн.
- Для `prompt_simulator_v*` всегда опирайся на реально последнюю версию из `System`, а не на захардкоженный номер.
- Структуру бери из ближайшего живого эталона, но не копируй его содержимое. Сначала совпадение секций, потом адаптация под задачу.
- Если канон и пользовательский запрос конфликтуют, явно выбери пользовательский запрос только когда он просит retcon/редизайн; иначе сохрани канон и отметь ограничение.

### Аватар-специфичные

- Для аватаров не дописывай характер, мотивацию, привычки, предпочтения или мировоззрение, если пользователь не задал их явно или они не следуют напрямую из входного файла.
- Для аватаров не оставляй формулировки вида "см. другой файл", "как в другом файле", "подробности в ..." и любые иные отсылки, требующие чтения соседних файлов для понимания самого аватара.
- Всё, что пользователь не описал явно и что не закреплено каноном входного файла, оставляй в нейтральной позиции без самовольной конкретизации.

### Навык-специфичные

- Если навык существует как файл в `Skills/`, не копируй его целиком — вставь summary и ссылку.
- Полная копия допустима только при модификации навыка для конкретного мира.

### Промпт-специфичные

- При адаптации промпта под другую модель сохраняй ВСЕ инварианты (два закона, щиток, recency zone, алгоритм, режимы Архиса).
- Модель-специфические блоки (антифабрикация, контроль длины, якорь формата) привязаны к архитектуре целевой модели и ДОЛЖНЫ быть переписаны, а не скопированы.
- При портировании не копируй model-specific комментарии из исходного файла — напиши новые для целевой модели.

### Контент

- Эротика, мрачность, power fantasy, комедия и кринж-намеренность допустимы только если это уже поддержано запросом или каноном файла.
- Не повышай рейтинг, жестокость, сексуализацию или психологическую тяжесть "для глубины", если задача этого не требует.

## Quality Bar by Artifact

| Артефакт | Минимальный результат |
|----------|----------------------|
| Аватар | Самодостаточное описание, понятная стартовая ситуация, нейтральность в чертах личности, если они не заданы. |
| Навык | Ограничения, стоимость/условия, понятная игровая функция, отсутствие формулы "может всё". |
| NPC / спутник | Роль в сцене, узнаваемый голос, связь с миром или аватаром, не только биографическая справка. |
| Мир | Играбельные правила, конфликты, ограничения, точки входа для симуляции. |
| Сценарий / эксперимент | Условия старта, участники, угрозы/давление, критерии развития сцены. |
| `prompt_simulator_v*` | Сохранены инварианты, формат валиден, model-specific блоки соответствуют целевой модели. |

## Output Workflow

1. Ответь на Quick Assessment (3 вопроса).
2. Загрузи нужные references по типу задачи.
3. Найди канон и эталоны подходящего масштаба.
4. Собери структуру будущего файла по живым примерам проекта.
5. Определи жанр и тон.
6. Напиши или переработай контент в стиле репозитория.
7. Проверь по Anti-patterns.
8. Проверь по Final Validation.
9. Только после этого сохраняй файл.

После сохранения проверь diff: правка должна соответствовать запросу, не содержать случайной перекодировки, массового переформатирования или потерянных секций.

## Reference Map

- `references/canon-and-file-discovery.md`: поиск канона, путей, последних версий и naming patterns.
- `references/examples-index.md`: индекс лучших живых образцов проекта, разделённый по масштабу и типу.
- `references/avatar.md`: аватары и world-integrated стартовые сетапы. Включает: масштаб, спутников, статы, стартовую сцену, жанровые шаблоны и механику-фишку.
- `references/skill.md`: навыки, способности, дары и механики силы. Включает: standalone vs встроенный, copy-paste навигацию, пассивные навыки, лотерейные системы.
- `references/character.md`: полные карточки персонажей, короткие NPC summary и спутники. Включает: каноничных персонажей, групповую динамику, школьные NPC-коллекции.
- `references/world.md`: миростроение и интеграция мира с аватаром. Включает: horror-миры, скрытую магию, лиминальные пространства.
- `references/story.md`: сценарии, эксперименты и игровые заготовки. Включает: Архис-режимы, experiment setup.
- `references/simulator-prompts.md`: эволюция `prompt_simulator_v*` без регрессий. Включает: JSON и Markdown форматы, автодетекцию версии, Model Porting Guide для адаптации под разные модели.

## Anti-Patterns (Не Делай Никогда)

| # | ❌ Ошибка | Контекст |
|---|-----------|----------|
| 1 | Характер от себя, когда не задан | Аватары |
| 2 | «Подробности в другом файле» | Аватары |
| 3 | Copy-paste навыка вместо ссылки | Навыки |
| 4 | Внешность = «красивый» без конкретики | Аватары, NPC |
| 5 | Все статы ★★★★★ | Аватары |
| 6 | Стартовая сцена = summary без атмосферы | Аватары |
| 7 | Мир = энциклопедия без играбельности | Миры |
| 8 | NPC без функции в сцене | NPC, спутники |
| 9 | Каноничный персонаж сглажен до безликого | NPC |
| 10 | Захардкоженная версия промпта | Simulator prompts |
| 11 | Потерянный инвариант при обновлении | Simulator prompts |
| 12 | Кроссовер = хаотичный список имён | Сценарии |
| 13 | Арена без ограничений и угроз | Эксперименты |
| 14 | Формулировки «может всё» без деталей | Навыки |
| 15 | Механика-фишка описана как навык с уровнями и активацией, хотя это привычка или поведенческий паттерн | Аватары |
| 16 | Адаптация под модель = copy-paste без переписывания model-specific блоков | Simulator prompts |
| 17 | Жанр не влияет на тон стартовой сцены: horror-аватар начинается весело, rom-com — с угрозы | Аватары |
| 18 | Новый лор добавлен как "давно известный факт", хотя его нет в источниках | Все |
| 19 | Файл стал длиннее, но не стал играбельнее | Миры, сценарии, аватары |
| 20 | Переписывание уничтожило локальные формулировки, по которым файл узнавался в проекте | Все |

## Final Validation Flow

Перед сохранением проверь:

1. Тип файла совпадает с запросом.
2. Путь и имя файла соответствуют паттернам проекта.
3. Масштаб файла соответствует масштабу запроса (не перегружай лёгкие, не обрезай тяжёлые).
4. Структура похожа на живые эталоны этого типа и масштаба.
5. Канон не сломан и новые факты не противоречат соседним файлам.
6. Тон и жанр соответствуют миру и запросу.
7. Стартовая сцена — это сцена с атмосферой, а не summary (для аватаров).
8. Спутники описаны полноценно (если есть).
9. Навыки не скопированы из существующих файлов без причины.
10. Каноничные персонажи сохранили узнаваемость.
11. Ни один пункт из Anti-Patterns не нарушен.
12. Для JSON:
    - файл валиден;
    - версия обновлена осознанно;
    - ключевые правила симулятора не деградировали.
13. Для Markdown:
    - XML-теги на месте;
    - форматирование корректно;
    - recency zone присутствует.
14. Для адаптации промпта под модель:
    - model-specific блоки переписаны, а не скопированы;
    - инварианты (два закона, щиток, режимы, recency zone) сохранены;
    - target-модель указана в заголовке/комментариях;
    - anti-fabrication и format anchors привязаны к реальным особенностям целевой модели.
15. Жанровый тон согласован: стартовая сцена, описание мира, тон NPC и динамика спутников работают на один жанр.
16. Diff чистый: нет случайной смены кодировки, лишних файлов, массового форматирования и правок вне задачи.

Если хотя бы один пункт не сходится, исправь до сохранения.
