# Canon and File Discovery

## Purpose

Используй этот reference перед созданием или переписыванием файлов в `Simulator Life`, чтобы не потерять канон, naming pattern и актуальные версии.

## Discovery Order

1. Посмотри, дал ли пользователь конкретный файл.
2. Найди похожие файлы того же типа в проекте.
3. Проверь соседние файлы того же мира или той же сущности.
4. Проверь вложенные каталоги (`Avatars/`, `Skills/`, `Skills/Avatars/`, `Avatars/For Leon/`).
5. Только потом формируй структуру нового документа.

## Where to Look First

### Root-level files

В корне проекта часто лежат:

- аватары: `*Avatar*.txt`
- сценарии/эксперименты: `*Scenario*.txt`, `*Experiment*.txt`
- старые или живые world-specific карточки

### Avatars/

Каталог `d:\Skvernost\Prompting\Simulator Life\Avatars` содержит:

- базовые аватары (`Leon_Avatar_Base.txt`)
- тематические аватары (`Leon_Avatar_DemonSlayer.txt`, `Leon_Avatar_ResidentEvil.txt`)
- NPC-аватары (`Ichi_Shinigami_Avatar_DemonSlayer.txt`)
- вложенный `Avatars/For Leon/` — специфические варианты для одного персонажа

### Skills

Каталог `d:\Skvernost\Prompting\Simulator Life\Skills` используй для:

- готовых навыков (`Дар Создания.txt`, `Ваучеры.txt`);
- навыков, привязанных к аватарам (`Skills/Avatars/Сердце Укротителя.txt`);
- проверки power scale.

### System

Каталог `d:\Skvernost\Prompting\Simulator Life\System` используй для:

- `prompt_simulator_v*.json` (JSON-формат, версии v3–v10)
- `prompt_simulator_v*_*.md` (Markdown-формат, начиная с v11; один промпт = несколько адаптаций под разные модели)
- Актуальные адаптации v11: `_opus.md` (Claude), `_gemini.md` (Gemini), `_deepseek.md` (DeepSeek)
- системных промптов
- вспомогательных шаблонов вроде `character_npc_summary_prompt_v1.txt`

## Naming Patterns

Придерживайся существующих имён проекта, если пользователь не указал иначе.

### Корневые файлы

- Аватар: `<Name>_Avatar_<World>.txt`
- Аватар с навыком: `<Name>_Avatar_<World>_Dar_<SkillName>.txt`
- Аватар с ролью: `<Name>_Avatar_<World>_<Role>.txt`
- Персонаж: `<Name>_Character_<World>.txt`
- Сценарий: `Arhis_Scenario_<ShortTitle>.txt`
- Эксперимент: `Archis_Experiment_<ShortTitle>.txt`

### Вложенные каталоги

- Навык (корень): `Skills\<SkillName>.txt`
- Навык (аватарный): `Skills\Avatars\<SkillName>.txt`
- Аватар (каталог): `Avatars\<Name>_Avatar_<World>.txt`

### System

- Промпт JSON: `System\prompt_simulator_v<N>.json`
- Промпт JSON с меткой: `System\prompt_simulator_v<N>-<label>.json`
- Промпт Markdown: `System\prompt_simulator_v<N>_<model>.md` (например: `v11_opus.md`, `v11_gemini.md`, `v11_deepseek.md`)

Если в проекте уже существует явный вариант имени для того же мира или той же серии файлов, выравнивайся по нему.

## Canon Rules

- Источник истины номер один: файл, который пользователь дал на вход.
- Источник истины номер два: соседние файлы того же мира.
- Источник истины номер три: уже существующие системные файлы проекта.
- Внешний канон франшизы вторичен. Он помогает только если проект не зафиксировал иное.

### Навыки и дубликаты

- Если навык уже существует как файл в `Skills/` или `Skills/Avatars/`, не копируй его содержимое в новый аватар.
- Впиши ссылку на существующий файл или краткий summary с пояснением, что детали — в файле навыка.
- Полная копия навыка в аватар допустима только если пользователь явно это просит или если навык модифицирован для конкретного мира.

## Create vs Rewrite vs Upgrade vs Port

### Create

- Найди 2-3 эталона того же типа.
- Сохрани стиль, но не копируй дословно.
- Не вводи жёсткие факты о мире, которых нет в каноне или запросе.

### Rewrite

- Сохрани исходный замысел.
- Не меняй имя, мир, базовые отношения, механику или стартовую ситуацию без явного запроса.
- Усиливай структуру и ясность, но не подменяй сущность.

### Upgrade

Для `prompt_simulator_v*`:

- найди последнюю реально существующую версию в `System`;
- используй её как базу, если пользователь не просит другой base file;
- создавай следующую версию только осознанно.

### Port

Адаптация `prompt_simulator_v*` под другую целевую модель:

- возьми существующую адаптацию как базу (например, `v11_opus.md`);
- сохрани все инварианты (два закона, щиток, режимы, recency zone);
- перепиши model-specific блоки (антифабрикация, якорь формата, контроль длины);
- НЕ копируй комментарии, привязанные к исходной модели;
- подробнее — в `references/simulator-prompts.md` (секция Model Porting Guide).

## Latest Simulator Prompt Rule

Не используй захардкоженное "vX -> vY".

Нужно:

1. Посмотреть, какие `prompt_simulator_v*.*` (и `.json`, и `.md`) есть в `System`.
2. Определить самую свежую по номеру или по явной задаче пользователя.
3. Строить новую версию от неё, а не от старой инструкции в голове.

**Правило автодетекции:**

```
Перед работой с prompt_simulator:
1. list_dir или grep System/ на prompt_simulator_v*
2. Определить максимальный номер версии
3. Определить формат: JSON (.json) или Markdown (.md)
4. Использовать как базу
```

## Minimal Discovery Checklist

Перед записью файла ответь себе:

- Какой файл или набор файлов задаёт канон?
- Какие 1-3 примера лучше всего показывают структуру?
- Куда именно должен быть сохранён результат?
- Не существует ли уже более свежая версия или соседний файл, который нельзя игнорировать?
- Нет ли уже готового навыка или мира, который можно подключить, а не дублировать?
- Если файл идёт в подкаталог — соответствует ли паттерн имени соседям?
