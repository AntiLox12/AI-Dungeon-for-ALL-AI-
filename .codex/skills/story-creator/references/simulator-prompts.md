# Simulator Prompt Guide

## Purpose

Используй для создания новой версии или переписывания `prompt_simulator_v*` внутри `Simulator Life`.

Сначала сверяйся с:

- `references/canon-and-file-discovery.md`
- `references/examples-index.md`

## Current Repo Reality

Не опирайся на старую инструкцию "создать `vX` из `vY`" с захардкоженным номером.

**Правило:** перед любой работой с промптом симулятора — определи последнюю реальную версию.

```
1. list_dir или grep System/ на prompt_simulator_v*
2. Определить максимальный номер версии
3. Определить формат: JSON (.json) или Markdown (.md)
4. Используй как базу, если пользователь не назвал другую
```

Для быстрой read-only проверки можно использовать:

```powershell
python .codex\skills\story-creator\scripts\project_inventory.py --project-root . --skill-root .codex\skills\story-creator --pretty
```

## Format Detection

Проект исторически использовал два формата. Определи формат базовой версии:

### JSON формат (v3–v10)

- Файл: `prompt_simulator_v<N>.json` или `prompt_simulator_v<N>-<label>.json`
- Структура: вложенные объекты и массивы
- Валидация: JSON-парсер должен пройти без ошибок
- Эталон: `prompt_simulator_v10-archis.json`

### Markdown формат (v11+)

- Файл: `prompt_simulator_v<N>_<target>.md`
- Структура: Markdown, часто с XML-envelope (`<system_prompt>`, `<identity>`, `<priorities>`, `<world_engine>`, `<npc_core>`, `<memory_protocol>`, `<algorithm>` и т.д.)
- Envelope зависит от порта: Opus/Gemini/Grok/ChatGPT используют `<system_prompt>`, DeepSeek в текущем дереве использует plain Markdown-заголовки.
- Комментарии: HTML-комментарии `<!-- ... -->` в начале файла, если они уже есть в базовом порте.
- Цель формата: оптимизация для конкретной модели (например, Claude Opus 4.6)
- Эталон: `prompt_simulator_v11_opus.md`

### Правило выбора формата

- Если пользователь не указал формат — используй формат последней версии.
- Если пользователь не просит реархитектуру — сохраняй envelope базового файла. Не добавляй XML только потому, что другой port его использует.
- Если пользователь просит перевод из JSON в Markdown или наоборот — это допустимо, но осознанно.
- Не смешивай форматы в одном файле.

## Upgrade Workflow

1. Найди все `prompt_simulator_v*.*` в `System`.
2. Определи формат и базу:
   - если пользователь назвал версию, работай от неё;
   - иначе используй самую свежую актуальную версию.
3. Выдели запрошенные изменения.
4. Сохрани ключевую структуру и порядок блоков, если пользователь не просит реархитектуру.
5. Обнови `_meta` / заголовок / комментарий осознанно.
6. Проверь валидность формата и сохранение envelope.

## Core Invariants: Do Not Regress

При любом улучшении не ломай следующие принципы:

### Два Абсолютных Закона

1. **Чёрный ящик:** Архис и его Аватар — чёрный ящик. Симулятор описывает мир вокруг них, но НИКОГДА не пишет за Игрока (ни действий, ни мыслей, ни эмоций, ни реплик).
2. **Тишина:** Финал ответа = действие мира. Точка. Никаких вопросов, советов, вариантов.

### Ядро философии

- Причинность > Нарратив.
- Данные > Фантазия.
- Формулы > Память.
- Инициатива мира > Пассивность.
- NPC действуют из СВОИХ мотивов, не ради Игрока.

### Режимы Архиса

- **Оболочка Пустоты:** невидимый наблюдатель, мир его не фиксирует.
- **Воплощённый Аватар:** физическое тело, включаются правила ресурсов и социалки.
- **Божественное Вмешательство:** чудо, знак, аномалия; NPC строят выводы в рамках своего мира.

### Щиток как источник истины

- Щиток обязателен в каждом ответе.
- Щиток_сейчас = Щиток_прошлого_хода ± обоснованные изменения.
- КОНТРОЛЬ ДАННЫХ: каждое поле проверяется явно.

### Антидрейф и память

- Сущности не удаляются — меняют статус.
- Приоритет источников: файл эксперимента → файл Аватара → данные проекта → master-file → прошлый Щиток → контекст чата.
- ПАМЯТЬ АРКИ тянется каждый ход.
- Рефлексия ловит упущенные детали.

### Структурные блоки (не теряй при обновлении)

- Приоритеты (P1-P4).
- Хроники Архиса / режимы присутствия.
- Механика NPC: слои глубины, архетип, речь, меж-NPC динамика.
- Отношения: 4 слоя (личное, социальное, практическое, скрытое).
- Внешность Аватара как социальный фактор.
- Эхо последствий: немедленные, отложенные, фоновые.
- Алгоритм генерации (5-7 шагов).
- Формат ответа: Рефлексия → Щиток → Текст → Финал.
- Механика сна и команды.
- Антизапрет и антиклише.

## High-Value Blocks to Protect

Особенно внимательно проверяй:

- `_meta` / заголовок с версией
- блоки приоритетов и жёстких правил
- механику щитка / state continuity / anti-drift
- ограничения формата ответа (лимит слов, порядок блоков)
- блоки про инициативу мира, причинность и финал ответа
- рефлексию (чеклист перед генерацией)
- recency zone (дубликация критических правил в конце для attention survival)

Если меняешь их, изменение должно быть явным и осмысленным.

## Versioning Rule

Не повышай номер версии автоматически просто потому, что был edit.

Осмысленный next version нужен, если:

- создаётся отдельный новый файл;
- пользователь явно просит новую версию;
- изменения достаточно крупные, чтобы отделить файл от базы.

Если пользователь просит правку текущего файла без отдельной версии, не навязывай новый номер.

## Structural Rules

### Для JSON-формата

- Сохраняй JSON-валидность.
- Не меняй структуру на Markdown или псевдо-JSON.
- Не ломай существующие ключи без необходимости.
- Если реорганизуешь блоки, убедись, что логика осталась читаемой.

### Для Markdown-формата

- Сохраняй envelope базового файла:
  - если есть XML-теги (`<system_prompt>`, `<identity>` и т.д.) — они остаются;
  - если файл plain Markdown — не превращай его в XML без явной причины.
- Не смешивай XML-теги с JSON-объектами.
- Используй `---` для разделения секций.
- HTML-комментарии `<!-- -->` для мета-информации и target-модели, если этот паттерн уже используется в выбранном порте.
- Markdown-форматирование: заголовки, таблицы, списки, code-blocks.

## Anti-Patterns (Не Делай)

| ❌ Ошибка | ✅ Как правильно |
|-----------|----------------|
| Захардкоженный номер версии | Автодетекция: сканируй System/ |
| Потерян блок при обновлении | Чеклист инвариантов перед сохранением |
| JSON-формат для v11+ | Markdown + XML-теги |
| Markdown-формат для v3-v10 | JSON |
| DeepSeek/plain Markdown port насильно переведён в XML | Сохрани envelope базового файла |
| Удалена recency zone | Критические правила дублируются в конце |
| Убран КОНТРОЛЬ ДАННЫХ из Щитка | Верификация каждого поля обязательна |
| Изменена структура Щитка без причины | Порядок полей стабилен между версиями |
| Добавлены советы Игроку в финал | Финал = действие мира. Точка. Тишина. |

## Model Porting Guide

### Когда использовать

Задача `port` — адаптация существующего `prompt_simulator_v*` под другую целевую модель. Например: взять `v11_opus.md` и создать `v11_deepseek.md`.

### Current Versions Table

| Файл | Целевая модель | Формат |
|------|----------------|--------|
| `System\prompt_simulator_v11_opus.md` | Claude Opus 4.6 | Markdown + XML-envelope |
| `System\prompt_simulator_v11_gemini.md` | Gemini 3.1 Pro Preview | Markdown + XML-envelope |
| `System\prompt_simulator_v11_deepseek.md` | DeepSeek V3.2 | Plain Markdown |
| `System\prompt_simulator_v11_grok420.md` | Grok 4.20 Reasoning | Markdown + XML-envelope |
| `System\prompt_simulator_v11_chatgpt55.md` | ChatGPT GPT-5.5 / GPT-5.5 Pro | Markdown + XML-envelope |

Таблица — снимок текущего дерева. Перед работой всё равно сканируй `System/`.

### Инварианты (НЕЛЬЗЯ менять при портировании)

Эти элементы идентичны во всех адаптациях:

- **Два абсолютных закона** (Чёрный Ящик, Тишина).
- **Режимы Архиса** (Хроники, Оболочка Пустоты, Божественное Вмешательство).
- **Щиток v11** (структура, порядок полей, КОНТРОЛЬ ДАННЫХ).
- **Recency Zone** в конце промпта.
- **Алгоритм генерации** (рефлексия → сцена → щиток).
- **Антидрейф** (запрет на OOC, советы, повторы).

### Model-Specific Blocks (НУЖНО переписывать)

Эти блоки привязаны к архитектуре целевой модели и **должны быть переписаны**, а не скопированы:

| Блок | Зачем | Пример (DeepSeek V3.2) |
|------|-------|------------------------|
| **АНТИФАБРИКАЦИЯ** | Модели с высокой инференцией додумывают канон | `НЕ придумывай факты, не выбирай за Аватар, не приписывай эмоции` |
| **ЯКОРЬ ФОРМАТА** | Некоторые модели склонны к расширению/сжатию | `Каждый ответ: описание мира → описание NPC-действий → Щиток` |
| **КОНТРОЛЬ ДЛИНЫ** | Разные модели имеют разные тенденции по длине | `Абзац описания = 2-4 предложения. Максимум = 350 слов текста` |
| **Target в заголовке** | Идентификация | `target="deepseek-v3.2"` |
| **Стиль формулировок** | Некоторые модели лучше реагируют на императив, другие — на описательный стиль | Зависит от модели |
| **Граница инструментов / поиска** | Модели с tool-use могут подмешать внешние факты | Для ChatGPT/Gemini/Grok явно запретить использовать search/tools как канон без ввода Игрока |
| **Thinking/reasoning правило** | Reasoning-модели могут начать объяснять внутренний ход | Рефлексия = короткий чеклист, не chain-of-thought |

### Чеклист портирования

1. Взять последнюю версию из `System/` (проверить через сканирование директории).
2. Скопировать базовую структуру.
3. Заменить `target` в заголовке.
4. **Переписать** model-specific блоки для целевой модели.
5. Проверить, что все инварианты сохранены (два закона, щиток, режимы).
6. Проверить, что recency zone присутствует.
7. НЕ копировать комментарии и примечания из исходного файла, если они привязаны к другой модели.
8. Проверить формат и envelope: XML-tagged Markdown остаётся XML-tagged, plain Markdown остаётся plain Markdown.

### Naming

`prompt_simulator_v<N>_<model>.md`

Примеры:
- `prompt_simulator_v11_opus.md`
- `prompt_simulator_v11_gemini.md`
- `prompt_simulator_v11_deepseek.md`

## Simulator Prompt Validation Checklist

Перед сохранением проверь:

- Базовая версия выбрана осознанно, а не по устаревшему шаблону.
- Формат файла правильный (JSON или Markdown) и валидный.
- Envelope базового Markdown-порта сохранён.
- `_meta` / заголовок обновлён только там, где это нужно.
- Все инварианты симулятора сохранены (два закона, режимы, щиток, антидрейф).
- Нет регрессии в сторону советов игроку или контроля Аватара.
- Если создаётся новая версия, её номер и путь названы последовательно.
- Recency zone присутствует в конце промпта.
- КОНТРОЛЬ ДАННЫХ присутствует в Щитке.
- Рефлексия присутствует в алгоритме генерации.
- Форматные правила (лимит слов, порядок блоков, тишина в финале) на месте.
- **Если это адаптация под модель:**
  - Target-модель указана в заголовке.
  - Model-specific блоки переписаны, а не скопированы.
  - Инварианты (два закона, щиток, режимы, recency zone) сохранены.
  - Граница инструментов/поиска и reasoning/thinking правило соответствуют целевой модели.
  - Нет комментариев/примечаний, привязанных к исходной модели.
